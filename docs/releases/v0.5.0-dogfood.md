## Dogfood findings and corrections

### Route/stop mismatch

The original validation example paired route 109 with Melbourne University,
which route 109 does not serve. The old candidate returned an unexplained empty
success. The corrected CLI validates membership through the documented
cross-mode `StopsForRoute` endpoint.

Post-fix evidence:

- Melbourne University + tram 109 exits nonzero, writes no partial JSON, and
  explains that the route does not serve the stop.
- `next 'Box Hill Central' --mode tram --route 109 --limit 3` returns three live
  route-109 departures.
- A served route with no current departures remains a valid empty result in the
  regression fixture.

### Fare contract

The Fare Estimate endpoint has `FareEstimateResultStatus` and
`FareEstimateResult`, but no generic PTV `status`. The old candidate fabricated
an empty version/health `0` from a nonexistent DTO field.

Post-fix human and JSON fare checks succeeded. JSON has exactly the two
endpoint-specific top-level keys and no lowercase generic `status`; non-success
fare result codes now return an error.

### Unsequenced route stops

PTV route data can mix positive stop sequences with upstream zero values. The
CLI continues to preserve `0` in JSON for compatibility, sorts those rows after
known positive sequences, and renders the human sequence as `-`. Route 109
revalidation observed 32 zero values across both directions and no human `0`
sequence labels.

### Credential source

An apparent status/check source difference was a sandbox/keyring-access
artifact. In the same execution context both commands used the same documented
environment → OS keyring → explicit env-file precedence.

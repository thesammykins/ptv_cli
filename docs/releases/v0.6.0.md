# ptv 0.6.0

This release adds a credential-free offline timetable foundation and bounded
static PTV v3 enrichment while preserving the existing live API contracts.
Local GTFS is authoritative for route topology, schedules, and planning. The
bundled v3 data is supplementary and deliberately excludes route and timetable
data so that a snapshot cannot make a user's route or departure information
stale.

## Highlights

- Adds validated GTFS Open Data generations with schema-v2 generation-local
  integer keys, namespaced public identifiers, Melbourne service-day handling,
  and local timetable/trip queries.
- Adds the `timetable` and `trip` commands plus GTFS-first search, stops, lines,
  station, next, disruptions, and route views.
- Normalizes Transport Victoria GTFS-Realtime trip updates, service alerts, and
  vehicle positions with deterministic freshness, cache, per-endpoint rate
  limiting, and degraded output behavior.
- Adds `ptv gtfs realtime` catalog/inspection support and `ptv auth opendata`
  credential management. Open Data credentials are independent from PTV
  Timetable API v3 credentials.
- Adds a reviewed v3 snapshot containing only slowly changing outlet and
  station-facility data. Route, direction, platform, and timetable data is
  never served from that snapshot.
- Keeps additive JSON source and warning fields, stable empty collections,
  source-qualified identifiers, and explicit attribution/disclaimer data.
- Applies global result limits after merged result sets, and reports freshness,
  update progress, and degraded live-source behavior honestly.
- Adds a pinned monthly snapshot-update workflow that can be enabled with
  repository secrets without putting credentials in the distributed binary.

## Upgrade and migration checklist

1. Install 0.6.0 and confirm the binary with `ptv version`.
2. Run `ptv gtfs status`. A database created by a pre-0.6.0 binary is reported
   as `legacy_reingest_required`; it is not silently treated as schema-v2.
3. Run `ptv gtfs update` once. This downloads and validates the current upstream
   feeds and publishes a new immutable schema-v2 generation.
4. Run `ptv gtfs check` and a credential-free smoke test such as
   `ptv --json timetable "Flinders Street"` or a local `ptv plan` query.
5. If using live v3 features, run `ptv auth status` and `ptv auth check`. If
   using GTFS-Realtime, configure the separate Open Data subscription key with
   `ptv auth opendata login`, then use `ptv auth opendata status` and
   `ptv auth opendata check`.

There is no SQL-in-place migration and no need to delete the existing database.
The first 0.6.0 update creates a schema-v2 generation alongside the old file;
the prior file remains available for rollback with an older binary. The active
generation is selected through `current.json`, with staging and the update
lease kept under the configured `PTV_DATA_DIR`. Do not copy a legacy SQLite
file into `generations/` or rename it as a v2 generation. If an update is
interrupted, the previous published generation remains usable; retry
`ptv gtfs update` after the process has exited.

The default data directory is unchanged. If `PTV_DATA_DIR` is set, use the
same directory consistently for all commands and ensure it is writable. No
credential migration is required: existing Timetable API keyring entries remain
valid, while the Open Data subscription key is stored separately.

## Compatibility and data-source behavior

- Existing JSON top-level fields remain available. `data_source`,
  `source_notice`, `freshness`, and `warnings` are additive. Consumers should
  tolerate unknown fields and empty collections/maps.
- GTFS integer keys are generation-local. Public GTFS identifiers are
  namespaced by feed mode; consumers should prefer `gtfs_*` fields and the
  source-qualified `id` rather than treating a GTFS ID as a PTV v3 numeric ID.
  The CLI does not fabricate PTV IDs for Open Data-only entities.
- Without a PTV v3 key, GTFS-backed search, lines, stops, station, timetable,
  trip, scheduled next departures, fare, and local planning remain usable after
  `ptv gtfs update`. The reviewed snapshot can provide selected static outlets
  and facility metadata, but never route or timetable data.
- With v3 but without an Open Data key, v3 live departures/disruptions remain
  available where supported; GTFS-Realtime next updates, Open Data alerts, and
  GTFS-Realtime vehicle positions are unavailable, so next falls back to
  scheduled GTFS output and live features degrade with an explicit warning.
- With Open Data but without v3, GTFS-Realtime can add trip updates, alerts, and
  vehicle positions where the upstream feeds expose them, but it does not
  provide the Timetable API's v3-only facilities, platform enrichment, or
  v3 disruption metadata.
- With neither credential, the CLI still provides the local static experience
  and clearly reports unavailable live sources. `ptv plan` is calculated locally;
  any optional live disruption overlay is best effort and is not required for a
  valid static journey.
- GTFS freshness checks are enabled by default where configured. Use
  `--no-update-check` to suppress the advisory check/auto-update path. JSON
  freshness states distinguish `current`, `stale`, `changed`, `updating`,
  `failed`, and `unknown`; a failed check is not reported as current.

## Data licensing and attribution

The bundled v3-derived data is distributed under the PTV Timetable API Data
Creative Commons Attribution 4.0 International Licence. Use of that data must
retain the requested attribution wording:

> Source: Licensed from Public Transport Victoria under a Creative Commons
> Attribution 4.0 International Licence.

Attribution, modification, disclaimer, and non-endorsement notices are included
in the repository `NOTICE` file and release archives. This project is
independent of PTV and does not imply PTV endorsement. The API data is provided
as-is; users are responsible for deciding whether it is suitable for their use.

## Known limitations

- GTFS and GTFS-Realtime coverage, identity quality, and freshness remain
  dependent on Transport Victoria's upstream feeds. The CLI surfaces upstream
  duplicate/non-increasing-trip warnings rather than silently repairing them.
- The Open Data service-alert catalog currently has known upstream feed
  coverage limitations by mode. Missing or unauthorized feeds degrade to an
  explicit warning rather than being presented as an empty authoritative feed.
- Live vehicle visibility is not guaranteed. A descriptor or position may be
  absent even when a vehicle is operating; any last-spotted result is labelled
  as such and is not presented as proof of current service.
- The bundled snapshot intentionally excludes route, direction, platform, and
  timetable data. It must be refreshed when the reviewed static enrichment
  source changes; it is not a replacement for the GTFS update lifecycle.
- Release archives are unsigned/not notarized for macOS; users may need to
  approve the downloaded binary according to their local security settings.
